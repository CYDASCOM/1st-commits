# GitHub 開発フロー完全ガイド (Macインストール対応)

MacへのGitインストールから、GitHubを用いたチーム開発の基本ワークフロー（CloneからMergeまで）を解説する手順書です。

---

## 1. MacにGitをインストールする

### 手順 ①：Xcode Command Line Tools を使用
Macで最も一般的な方法です。ターミナル（Terminal.app）を開き、以下のコマンドを実行します。

```bash
# インストールコマンドの実行
xcode-select --install

```

**【実行後の流れ】**

1. 画面に確認ダイアログが表示されるので「インストール」をクリックします。
2. 利用規約に同意し、ダウンロードとインストールが完了するのを待ちます。

### 手順 ②：Homebrewを使用する場合 ＆ 初期設定

パッケージマネージャー（Homebrew）を使ってGitを管理・アップデートしたい場合は、以下のコマンドを使用します。

```bash
# Gitのインストール
brew install git

```

#### 必須の初期設定

インストール完了後、GitHub上でコミットした際に「誰が変更したか」を正しく記録するために、以下のユーザー名とメールアドレスの設定を必ず行ってください。

```bash
# グローバルユーザー名とメールアドレスの紐付け
git config --global user.name "Your Name"
git config --global user.email "example@cydas.com"

```

## 2. 基本のGitワークフロー (Clone ～ Merge)

### Step 1: Clone (リポジトリの複製)

GitHub上にあるリモートリポジトリのソースコードを、手元のローカルPCにコピーして作業環境を構築します。

```bash
# 指定したリポジトリのURLからローカルに完全コピーを作成する
git clone https://github.com/CYDASCOM/1st-commits.git

```

```bash
# クローン完了後、新しく作成された対象プロジェクトのディレクトリに移動する
cd 1st-commits

```

### Step 2: Branch (ブランチの作成)

メインのソースコード（`master` ブランチや `main` ブランチなど）を直接書き換えて壊してしまわないよう、機能追加やバグ修正ごとに専用の「作業スペース（ブランチ）」を作成して移動します。

```bash
# 新規ブランチの作成と、そのブランチへの切り替えをワンラインで同時に実行する
git checkout -b trial/[あなたのローマ字苗字]

```

*※ 開発現場では `feature/login-fix` や `bugfix/issue-12` のような分かりやすい識別名を付けるのが一般的です。*

### Step 3: Commit (変更の記録)

作業スペース内で行ったファイルの修正内容を、履歴（スナップショット）としてローカル環境に保存・確定させます。

```bash
# 現在のディレクトリ以下にあるすべての変更・新規ファイルをまとめてステージングに追加する
git add .

```

```bash
# 変更内容の要約文（コミットメッセージ）を引用符内に付与して履歴を確定させる
git commit -m "○○機能を実装"

```

### Step 4: Push (リモートへ送信)

手元のPC（ローカルリポジトリ）で確定させたコミット履歴を、GitHub上のサーバー（リモートリポジトリ）にアップロードして、チームに共有できるようにします。

```bash
# 接続先サーバー（origin）に対して、手元で作成した指定ブランチをそのまま送信する
git push origin trial/[あなたのローマ字苗字]

```

*※ `origin` は、最初に `git clone` を行った際に自動で設定されるデフォルトのリモート先サーバーの識別名です。*

### Step 5: Pull Request (レビュー依頼)

GitHubのWebブラウザ画面を開き、自分が変更したブランチを `main` ブランチに合流させるための「Pull Request (PR)」を作成します。

1. **PRの作成**: GitHub上で「Compare & pull request」ボタンを押し、タイトルや具体的な変更内容を記述します。
2. **レビューの依頼**: チームメンバーをレビュアーに指定し、コードをチェックしてもらいます。
3. **承認 (Approve)**: 指摘事項があれば修正コミットをさらに重ね、最終的に承認をもらいます。

### Step 6: Merge (成果の統合)

レビューが完了し、承認（Approve）が得られたら、いよいよ本番のコードに成果を統合します。

1. **GitHub上でのマージ**: Web画面上で「Merge pull request」ボタンを押し、`main` ブランチへ合流させます。
2. **ローカル環境の整理**: 役割を終えたブランチから戻り、手元のメイン環境を最新状態に同期します。

```bash
# 役割を終えたブランチから、基準となる大元のメインブランチへ切り替えて戻る
git checkout main

```

```bash
# リモートでマージされ新しくなった最新のmainブランチの内容をローカルに一括同期する
git pull origin main

```

---

## 3. 安全で確実な開発のための3大原則

* **必ずブランチを切る**: 直接 `main` をいじらないことが鉄則です。
* **コミットは細かく**: 何をしたか後から見追いやすいメッセージを心がけます。
* **作業開始前には必ずPull**: 他のメンバーの変更と衝突（コンフリクト）するのを防ぐため、毎日の作業開始前には必ず最新の `main` をローカルに取り込みましょう。
